Arrays vs. Hashes

First I'll talk a bit about arrays and then I'll get into hashes. Basically arrays are numbered lists where each number is associated with a slot that acts like a variable. Each slot can be assigned a value such as an integer, a string, a float, etc. Below is a simple example of an array.

            simple_array = ['first_entry', 'second_entry', 'third_entry']

Unlike arrays, hashes are not ordered and instead are lists of keys and values associated with each key. For example you might use a hash to keep track of countries and which continent they are on. The name of the country would be the key and the value would be the continent it is on. Below is an example of a hash like that.

            simple_hash = {
            'USA' => 'North America',
            'Colombia' => 'South America',
            'Cambodia' => 'Asia',
            'France' => 'Europe'
          }

As you might have figured out arrays are best when you want your list to be in numeric order and when there is only one value per entry that you'd like to add such as players on the 49ers. Hashes are better when you have a list where you want to associate a key with a value such as players on the 49ers AND their position--Note in this example the key value would be the player's name and the value would be the player's position because key values must be unique.

Why is pair programming important to DBC?

Pair Programming, as far as I can tell (I'm only in week 3) is integral aspect to DBC. This makes a lot of sense to me because DBC emphasizes learning from your peers and pairing is a great way to do this. Furthermore, DBC stresses the process of teaching to cement your understanding of a concept--which defintiely happens in pairing sessions. DBC also uses pairing sessions as a way to build engineering empathy within its Boots.

Comparison of Driver and Navigator Roles

I see the role of the Navigator as the teammate who keeps a higher level view of the project or problem and might even keep an eye on what is coming up ahead. They can also perform research on a topic if the driver has stuff to work on. For remote pairing sessions the navigator can also be in charge of running the code on their location machine periodically. I see the driver as the person who is more in the trenches of the code working on the syntax and just putting everything down in the Text Editor. If stuck with something the driver can also just test some ideas out while the navigator reads about the issue online.
